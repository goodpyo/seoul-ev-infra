# 서울시 전기차 충전 인프라 분석 — 발표 핵심 정리

> 데이터 기준: 전기차 등록 현황(2026.04) · 충전소 설치현황(2026.05.12)
> 작성 메모용 — 발표 자료는 추후 제작

---

## 1. 한 줄 핵심 메시지

> **"서울의 충전 인프라는 평균적으로 충분하다. 그러나 행정동 단위로 보면 충전 사각지대가 숨어 있다."**

자치구 단위 평균은 모든 구가 '충분'이라 문제가 안 보이지만,
행정동 단위로 쪼개면 **16개 동(3.7%)** 에서 부족이 드러난다 — 이것이 본 분석의 핵심 발견.

---

## 2. 서울 전체 현황 (총량)

| 항목 | 수치 |
|---|---|
| 전기차 등록 | **111,761대** |
| 충전기 총계 | **70,941기** (급속 6,019 / 완속 64,922) |
| 평균 부족지수 | **1.58** (충전기 1기당 전기차 대수) |

→ 총량으로는 충전기 1기당 전기차 1.58대. 절대량은 충분한 편.

---

## 3. "평균의 함정" — 단위에 따라 결론이 뒤집힌다

### 자치구 단위
- 25개 구 부족지수 **1.1 ~ 2.7** → **전부 '충분'**
- 가장 높은 구조차 강남구 2.7 (≤5 = 충분)
- 자치구 지도만 보면 "서울에 문제 없음"으로 오해

### 행정동 단위 (전기차 등록 427개 동 기준)

| 상태 | 동 수 | 비율 |
|---|---|---|
| 충분 | 373개 | **87.4%** |
| 보통 | 38개 | 8.9% |
| 부족 | 13개 | 3.0% |
| 매우 부족 | 3개 | 0.7% |
| 충전소 없음 | 0개 | 0.0% |

→ **부족 이상 = 16개 동 (3.7%)**. 평균에 가려진 사각지대.

---

## 4. 강남구 사례 — 평균의 함정을 보여주는 대표 예시

- **자치구 평균**: 전기차 15,631대 / 충전기 5,728기 → 부족지수 **2.7 (충분)**
- 그러나 동별 편차가 큼:
  - 개포1동: 부족지수 **16.8 (부족)** ← 같은 강남이지만 부족
  - 도곡2동: 부족지수 **0.8 (충분)**
- 강남구 전체로는 충분해 보여도, 동 단위로는 부족 지역이 존재

> 발표 포인트: "강남도 동별로 보면 부족한 곳이 있다" → 자치구 평균의 한계를 직관적으로 전달

3. 중곡1동 문화복지센터 (능동 126) — 완속 3기

**둔촌1동(1위)과의 차이**: 둔촌1동은 올림픽파크포레온 단지 내 전용 충전기가 별도 존재할 가능성이 있어 "공공 인프라 부족"으로 해석해야 하지만, **중곡1동은 일반 주거지역으로 대단지 아파트가 없고** 주민센터·절·복지센터에 완속 5기가 전부. 급속 충전기 0기.
→ **진짜 공공 충전 인프라 사각지대 — 발표에서 가장 확실한 사례로 제시 가능**
>
> ❌ **제외**: 강동구 **상일동**(구 표기상 부족지수 20.5)은 전기차 41대뿐인 **미세 잔여 버킷**.
> 인근 상일제1동(607대/366기=충분)·상일제2동(157대/128기=충분)은 충분 →
> 상일 지역 전체는 부족하지 않아 위기 지역에서 제외함.

## 6. 지표 정의 (발표 시 설명용)

- **부족지수 = 전기차 등록 수 ÷ 충전기 수** (충전기 1기가 감당하는 전기차 대수)
  - 값이 클수록 부족
  - 매우 부족(>20) / 부족(>10) / 보통(>5) / 충분(≤5)
- **충전편의지수 = (급속×3 + 완속) ÷ 전기차 × 100** (급속에 가중치)

## 7. 정책 시사점 (결론 방향)

1. **자치구 단위 정책의 한계** — 구 평균으로는 사각지대를 못 잡음. 행정동 단위 접근 필요.
2. **우선 투자 16개 동** — 특히 둔촌1동·중곡 일대·대치1동(고수요)에 충전기 우선 확충.
3. **수요 규모 함께 고려** — 부족지수가 같아도 대치1동(3,428대)처럼 절대 수요가 큰 곳을 우선.

## 8. 데이터 신뢰도 검증 결과 (중요 — 질문 대비)

행정동 단위 부족 데이터가 파싱 오류로 왜곡됐는지 **GPS 실측으로 직접 검증함.**

### 검증 방법
- 충전기 GPS 좌표를 행정동 경계 폴리곤에 점-포함(point-in-polygon) 검사 → 실측 충전기 수 산출
- 파이프라인 집계값과 대조

### 결과
- **TOP 부족 동 15곳 중 GPS 검증 가능한 14곳 모두 충전기 수 정확히 일치(±1)** → 부족 순위는 신뢰 가능
- 단, 두 가지 정밀도 한계 발견:

**(1) 법정동명 라벨 충전기 1,043기(전체의 1.5%) 미배정**
- 원인: 대부분(987기) GPS 없는 보완 데이터가 `둔촌동·반포동` 등 **법정동명**으로 라벨 → 행정동(둔촌1동 등)에 자동 배정 안 됨
- 영향: 일부 동의 부족이 **과장**될 수 있음 (절대 과소평가는 아님 — 부족이 실제보다 약하게 나올 일은 없음)
- 등급 변동 가능 동: 용산2가동(부족→충분?), 반포본동·독산4동·신월6동(부족→보통?)
- **둔촌1동·대치1동·중곡 일대·개포1동 등 11곳은 보정해도 부족 유지 → 확실**

**(2) 상일동 미세 버킷**
- `상일동`(41대), `상일제1동`(607대), `상일제2동`(157대)이 별도 버킷으로 공존 (파싱 오류 아님)
- `상일동` 41대 버킷만 부족지수 20.5로 튀어 보이나, 상일 지역 전체는 충분 → TOP에서 제외

### 지표 자체의 한계
- 부족지수는 **완속 충전기(아파트 등)까지 1기로 동일 카운트** → 실제 체감 부족과 차이 가능
- 충전기 70,941기 중 **완속이 64,922기(91.5%)** → 급속만 따로 보면 부족 지역이 더 늘어날 가능성

### 후속 개선 제안
- 법정동→행정동 매핑 보강(GPS 없는 보완 충전기는 주소 지오코딩) → 1.5% 손실 회복
- **급속 충전기 한정 부족지수** 별도 산출 → 더 현실적인 사각지대 도출

### 분석 대상 범위 (질문 대비)

**본 분석은 "공용(외부 개방형) 충전기"만 대상으로 한다.**

| 포함 O | 포함 X |
|--------|--------|
| 공영주차장 충전기 | 아파트 입주민 전용 충전기 |
| 마트·백화점 공용 충전기 | 회사 직원 전용 충전기 |
| 도로변 급속 충전소 | 개인 가정용 충전기 |
| 공공기관 개방 충전기 | |

- 한국환경공단 API와 전기차 충전소 설치현황 엑셀 모두 **공용 등록 충전기**만 수록
- 아파트 지하주차장 충전기는 대부분 입주민 전용이라 공단 DB에 미등록 → 분석에서 누락

> **둔촌1동(1위) 해석 주의**: 올림픽파크포레온(12,032세대)은 단지 내 전용 충전기가 별도 존재하므로,
> "공공 충전 인프라 부족"이지 "충전기 자체가 없는 것"과는 다름.
> 대규모 신축 단지가 밀집한 동은 실제보다 인프라가 부족하게 나타날 수 있음.

**발표 권장 표현**:
> "본 분석은 한국환경공단에 등록된 공용 충전 인프라를 기준으로 하며, 아파트 단지 내 입주민 전용 충전기는 분석 범위에서 제외됩니다."

## 9. "내 주변 충전소 — 실시간 현황" 미작동 원인 (질문 대비)

`getChargerStatus`(공공데이터포털 개방 API)로 실시간 충전 상태를 받아오려 했으나 표출 실패.
직접 API를 여러 차례 테스트해 원인을 규명함.

### ❗ 결론: "일 1,000건 한도 초과"가 원인이 **아님**
- 응답 코드가 `resultCode: "00"` (**정상**) — 할당량 초과 코드(`22`)나 인증 오류(`30`)가 아니었음
- 할당량 초과면 즉시 에러 응답이 오지만, 실제 증상은 **응답 지연/타임아웃**으로 양상이 다름
- (참고) 개발계정 일 1,000건 한도는 존재 → 다수 사용자 대상 서비스라면 별도 제약이지만, 현재 미표출의 직접 원인은 아님

### 실제 원인: 개방 API의 성능·안정성 한계

**(1) 응답이 극도로 느림 (호출당 약 21초)**
- 단일 충전소 조회에 15~25초 소요. 앱 타임아웃(초기 5초)에 전부 걸림 → 25초로 상향했으나 한계
- 실시간 웹 표출(여러 충전소 동시 조회)에는 사실상 부적합한 속도

**(2) 동시 요청을 못 견딤**
- 충전소 150곳을 동시(8스레드) 조회 → **100% 타임아웃**
- 부하가 쌓이면 단일 순차 조회조차 응답 불능 상태가 됨 (테스트 중 확인)

**(3) 등록된 실시간 데이터도 제한적**
- API가 응답할 때도 상당수 충전소가 `totalCount: 0`(실시간 기록 없음) 반환
- 운영사별 편차 큼. 실시간 제공 확인 운영사 예: **ST**(에스트래픽), **ME**, **CR**, **KO** 등
- ※ 정확한 커버리지 비율은 API가 너무 불안정해 신뢰성 있게 측정하지 못함

### ev.or.kr에서는 왜 실시간이 잘 나오나?
- ev.or.kr은 **한국환경공단 직영 사이트** → 충전사업자와의 **내부 실시간 연계 데이터**를 사용
- 우리가 쓴 **공개 개방 API(`getChargerStatus`)는 그 내부 데이터와 별개**로, 속도·안정성·커버리지가 떨어짐
- 즉 "데이터가 세상에 없는 것"이 아니라, **공개 API 경로의 품질이 내부 시스템에 못 미치는 것**

### 앱에 적용한 처리
- 타임아웃 상향(25초), 동시 요청 수 축소(5스레드)
- 카드에 상태별 메시지 분기: 🟡 "조회 실패(재시도 가능)" / ⬜ "실시간 미제공"
- 충전소 **설치 현황(정적 데이터)은 100% 정상 표시** — 실시간 가용 여부만 제한적

### 한 줄 요약 (발표용)
> "실시간 충전 현황은 **공개 개방 API의 응답 속도(~21초)·안정성 한계**로 표출이 어려움.
> ev.or.kr은 환경공단 **내부 실시간 연계**를 쓰기 때문에 가능한 것이며,
> 우리가 접근 가능한 공개 API는 그 품질에 못 미침. (개발계정 호출 한도 문제가 아님)"
